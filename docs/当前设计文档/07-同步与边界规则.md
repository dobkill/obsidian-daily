# 07. 同步与边界规则

## 1. 基本原则

所有页面共享同一套任务数据，因此不存在“某个页面保存一份独立任务”的概念。

数据变更采用“先写文件、后刷新视图”的顺序。每次新增、修改、删除或完成状态变更都会进入串行写入队列，目标 JSON 写入成功后才触发页面重新渲染。

## 2. 新增任务

### 2.1 单次任务

创建后只生成一个发生日期。

### 2.2 重复任务

创建后会在同一条 Task 内记录多个发生日期。

每日 / 每周重复必须提供以下任一条件：

1. 重复次数。
2. 重复结束日期。

自定义重复则必须提供明确的 `occurrenceDates` 日期集合；格式化文本导入中对应 `repeat:custom` 与 `dates:...`。

## 3. 修改任务

### 3.1 修改单次任务

直接更新当前任务。

### 3.2 修改重复任务

当前版本删除“过去不可变 + 历史切片”规则，改为：

1. 一条重复任务始终只有一条系列记录。
2. 系列编辑直接更新系列级标题、描述、重复规则和默认时间。
3. 系列编辑不会自动生成历史任务切片。
4. 若某次发生需要与系列不同，改为在该日期写入 `occurrenceOverrides`。

### 3.3 修改单次发生

从今日任务页或周任务图的实例上下文进入时，默认进入“编辑本次”。

规则：

1. 不修改系列级字段。
2. 写入对应日期的 `occurrenceOverrides`。
3. 当前版本支持单次覆盖的字段为：`title`、`description`、`startTime`、`endTime`。
4. 时间冲突校验以最终展开后的实例为准。
5. 今日任务页与周任务图都会读取覆盖后的实例内容。

## 4. 删除任务

### 4.1 删除单次发生

可删除任意日期的单次发生。

### 4.2 删除整个任务

删除整个任务时直接删除整条系列；若其下挂有真实子任务，则把子任务重新挂回父级组合任务或顶层。

## 5. 完成任务

### 5.1 完成本次发生

允许完成或取消完成任意日期的单次发生。

可修改的字段只有当前发生的：

1. 完成状态。
2. `completedAt`

### 5.2 到本次为止结束重复

会把系列中从开始到“本次”为止的发生全部标记完成，并移除本次之后的未来发生。

## 5.3 组合任务时间边界

组合任务必须有时间范围。轻量子任务若有时间，必须在组合任务时间范围内；挂入组合任务的真实子任务必须在同一天对应组合任务实例的时间范围内。

## 6. 热度图统计边界

热度图只统计：

```text
实例 completed === true
并且 completedAt 存在
```

任务计划日期不决定热度图颜色。

## 7. 时间冲突边界

若任务同时具备：

1. `startTime`
2. `endTime`

则在同一天内必须通过时间冲突校验。

不允许：

1. 结束时间早于或等于开始时间。
2. 与同一天已有排期任务重叠。

补充：

1. 新建、编辑和导入任务时，若启用自动冲突处理，会自动寻找同日可用的 1 分钟空档占位。
2. 组合任务容器与其下级任务允许时间范围重叠，但下级任务必须落在容器时间范围内。
3. `timeSlotMinutes` 当前主要用于自动排程吸附，不是所有手工录入的强制校验条件。

## 8. 数据目录切换边界

设置页修改数据目录后：

1. 插件会先检查目标目录是否为 Vault 内相对路径。
2. 如果目标目录已有插件数据，且 `config.json`、`projects.json`、`project-pages.json`、`note-task-index.json`、`write-history.json` 与 `tasks/*.json` 的现有文件都能按当前结构解析，则加载目标目录数据。
3. 如果目标目录不存在、为空，或现有插件数据格式错误，则用当前内存数据在目标目录写入新的插件数据文件。
4. 如果目标目录包含非插件文件，仍会拒绝使用，避免覆盖用户自己的文件。

## 9. 写入失败边界

1. 数据文件读取失败时，插件进入只读保护，不再自动覆盖异常文件。
2. 单次写入失败会让当前操作失败并显示错误；下一次保存仍会继续尝试写入，不会因为队列前一次失败而被永久跳过。
3. 插件卸载时会等待已进入队列的写入完成，并在非只读状态下额外写入一次当前数据快照。

## 10. 启动与入口刷新边界

1. 插件启动时会先按当前设置的数据目录加载一次文件，并在 workspace ready 后再补一次回读。
2. 点击“项目总览”和“今日任务”入口时，不会阻塞当前页面打开；会先显示缓存，再在回读完成后刷新页面。
3. 启动期如果 Vault 缓存还没索引到目标目录或 `tasks/*.json`，插件会回退到磁盘适配器直接读取，避免误判为空目录。
4. 若数据目录在启动期已经存在，即使 Obsidian 内部再次触发目录创建，也应直接忽略“目录已存在”场景，不再把它当作初始化失败。

## 11. 全局笔记任务同步

当前版本支持扫描 Markdown 中的项目任务块：

```md
<!-- pm:start -->
- [ ] 写解析器 @2026-05-18 09:00-10:30 #plugin
<!-- pm:end -->
```

规则：

1. Obsidian workspace ready 后后台扫描，不阻塞页面打开。
2. 命令面板中的“扫描全库项目任务块”可手动触发全库同步。
3. Markdown 文件修改后会延迟触发单文件同步。
4. 文件 hash 未变化时跳过解析。
5. 解析结果写入插件任务模型，并用 `sourceLinks` 记录来源。
6. 索引缓存写入 `note-task-index.json`。
7. 来源消失时先标记为 missing，不直接删除任务。

## 12. 自动排程边界

自动排程目前作为显式操作提供。

规则：

1. 默认处理当天已排期、未完成、未锁定任务。
2. 当前实现向后顺延冲突任务。
3. 甘特图锁定的任务不会被移动。
4. 移动记录写入 `write-history.json`。
5. 自动排程不能作用于过去日期。

## 13. 已删除旧逻辑

1. 已删除“历史切片 / 活动切片”双记录模型。
2. 已删除“过去日期不可编辑”的强限制。
3. 旧版本的数据加载后会把逾期未完成的单次任务自动顺延到今天，并寻找当天 1 分钟空档。该逻辑已删除。
