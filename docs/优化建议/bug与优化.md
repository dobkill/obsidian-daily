## 根本原因

1. 项目页“导出 Markdown”生成的是 `+ 任务：标题` / `+ 组合：标题`，冒号后没有空格；旧解析器却要求冒号后必须有空格，导致导出的计划行被直接判为“无法识别的行”。
2. 项目页“导出 Markdown”会按任务真实字段导出计划文本；普通任务允许没有开始/结束时间，因此导出的普通任务可能只有 `@YYYY-MM-DD`。
3. 项目页“批量导入”只允许“新任务计划复杂 Markdown”，但旧判定把 planned 格式等同于“每条任务都有完整 `@日期 开始-结束`”。只要导出文本里有未排期普通任务，就会被误判为极简 Markdown 或不完整格式。
4. 文档写成“创建或覆盖任务必须提供完整时间段”，没有说明代码实际支持普通任务只写日期，也没有明确组合任务/子任务才必须完整时间。

## 解决方案

1. 解析 `+ 任务：/+ 组合：/+ 子任务：` 时，冒号后空格改为可选，兼容导出文本和手写文本。
2. 解析时记录每条任务来自 `+` 计划语法，还是 `- [ ]` 今日完成语法。
3. planned 格式识别改为基于 `+` 计划语法，而不是基于是否有完整时间段。
4. 保留必要业务校验：普通任务必须显式写 `@YYYY-MM-DD`；组合任务和组合子任务必须完整写 `@YYYY-MM-DD HH:mm-HH:mm`。
5. 同步更新 Markdown 格式文档、当前设计文档和内置导出说明。

## 执行结果

已按方案修改解析/预览逻辑和相关文档。项目导出的 `+ 任务：标题` / `+ 组合：标题` 行可被识别；普通未排期任务再次批量导入时，会被识别为新任务计划复杂 Markdown；若组合任务或子任务缺少时间，会显示明确的阻断原因。
