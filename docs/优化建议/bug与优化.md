## 已落实的相关性分析与修改方案

### 任务相关性

1. “轻量项”与真实子任务并存，是数据格式、今日显示、周任务图和项目导入不一致的共同原因。统一为“独立普通任务挂入组合任务”后，完成记录、重复规则、看板、甘特图和思维导图状态都只需要维护一份任务数据。
2. 项目 Markdown 导出与项目批量导入必须使用同一种语义：复杂 Markdown 只描述项目计划和视图状态，不携带执行记录；执行记录只由数据迁移 JSON 负责。
3. 热力图和周任务图都是信息密度问题：热力图需要铺满固定底部宽度，周任务图的组合子任务需要只保留标题，避免卡片过高。

### 深层原因

1. 旧模型中 `Task.subtasks` 表示轻量项，真实子任务又通过 `viewState.mindmap.parentTaskId` 表示，导致组合任务进度、导出内容和视图展示有两套来源。
2. 项目导入弹窗限制的是 `markdown-planned`，但预览曾返回通用 Markdown 类型，导致导出的复杂 Markdown 在项目导入模式下被误判为不支持格式。
3. 项目 Markdown 曾导出完成日期，计划文本和执行记录混在一起，修改后再导入容易覆盖或污染完成状态。

### 完整方案

1. 数据模型：不再从 `subtasks` 创建轻量项；子任务只能是普通任务，并且只能挂入组合任务。
2. 导入导出：复杂 Markdown 支持 `+ 子任务：`、`board`、`gantt`、`deps`、`mindmap`，不再导出完成记录；项目页批量导入严格匹配当前项目名，同名任务覆盖计划字段并保留既有完成状态。
3. 数据迁移 JSON：继续作为唯一完整迁移格式，保留项目、任务、视图状态和执行记录；组合子任务通过独立任务的父级关系表达。
4. 今日极简 Markdown：只导出今天未完成的真实任务；组合容器下的未完成项按子任务导出。
5. UI：任务弹窗将“组合项”改为“子任务”，提供“添加子任务”；周任务图子任务使用紧凑标题展示；热力图列宽均分并铺满底部宽度。
6. 文档：同步更新当前设计文档、Markdown 格式说明和设置页/快速记录页内置示例。
